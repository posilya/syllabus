# Общее описание проекта и бизнес-процессы
## 1. Назначение документа

Документ описывает общий контекст проекта, текущий и целевой процессы работы с расписанием.
## 2. Краткое описание проекта

Система предназначена для создания, редактирования и просмотра расписания учебного заведения.

В первой версии сервис разрабатывается для колледжа УКСАП, но должен проектироваться с возможностью дальнейшего применения в других учебных заведениях.

Основная проблема текущего процесса заключается в том, что расписание ведётся в Excel и вручную публикуется на сайте колледжа.

## 3. Бизнес-цели и бизнес-требования

### 3.1. Бизнес-цели

Основные бизнес-цели проекта:

1. Упростить процесс создания и изменения расписания учебного заведения.

2. Сократить количество ручных действий при работе с расписанием.

3. Снизить количество ошибок, возникающих при составлении и публикации расписания.

4. Обеспечить студентам и преподавателям доступ к актуальному расписанию.

5. Упростить поиск расписания по группе, преподавателю и аудитории.

6. Создать основу для дальнейшего развития сервиса, включая уведомления, экспорт расписания и интеграции с другими системами.
### 3.2. Бизнес-требования

BR-001. Сократить ручные действия при создании и публикации расписания.

BR-002. Обеспечить пользователям доступ к актуальному расписанию.

BR-003. Система должна обеспечивать снижение количества ошибок за счёт проверки конфликтов расписания.

BR-004. Упростить просмотр расписания для студентов и преподавателей.

BR-005. Обеспечить возможность централизованного управления расписанием и справочниками.
## 4. Текущий процесс AS-IS

### 4.1. Описание текущего процесса

1. Составитель расписания создаёт или изменяет расписание в Excel.
2. Расписание проверяется вручную.
3. Составитель расписания публикует расписание на сайте колледжа.
4. Студенты и преподаватели открывают сайт и просматривают расписание.
5. При изменении расписания процесс повторяется.

### 4.3. Проблемы текущего процесса

- расписание ведётся вручную;
- при составлении расписания возможны ошибки;
- публикация изменений зависит от ручных действий;
- пользователи могут видеть неактуальную версию расписания;
- отсутствуют автоматические уведомления об изменениях.
## 5. Целевой процесс TO-BE

### 5.1. Описание целевого процесса

1. Редактор расписания входит в систему.
2. Редактор создаёт или изменяет занятие через веб-интерфейс.
3. Система сохраняет изменения.
4. При наличии конфликта система показывает предупреждение.
5. После сохранения расписание становится доступным для просмотра.
6. Студенты и преподаватели просматривают актуальное расписание без авторизации.
